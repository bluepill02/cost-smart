# Ad-block recovery fix and floating chat widget replacement

The change fixes a dead-code bug where `SoftAdBlockMessage` was imported in the root layout but never rendered, and replaces the non-functional Botpress chat integration (which depends on an unconfigured external bot URL) with a self-contained floating chat widget that uses client-side pattern matching to surface financial advice and calculator links. The ad-block notice is now rendered inside the `<Providers>` wrapper, which it requires for `useProStatus` context. The chat widget ships as a single 309-line client component with no external dependencies beyond `lucide-react` and `next/link`.

**Watch for:** z-index collision between the chat widget (z-40) and the ad-block banner (z-50) at `bottom-right` positioning (confirmed); the Botpress loader remains in the layout, producing a dead integration alongside the replacement widget (confirmed); no test coverage for either addition (confirmed).

## High-level view

The ad-block recovery fix is a one-liner that renders the already-imported component. Placing it inside `<Providers>` is necessary because it calls `useProStatus`, which reads from `ProStatusContext`.

The floating chat widget is a fully client-side pattern matcher ‚Äî no API calls, no secrets, no server interaction. It classifies user queries by intent (home loan, salary, investment, etc.) via regex, generates canned advice paragraphs, and appends links to the site's existing calculator pages.

The old Botpress integration (`BotpressLoader`) is still rendered in the layout below `</Providers>`. It's inert at runtime (env var unset), but leaves dead code and two competing "chat" concepts in the codebase. If someone sets the env var, two chat widgets appear simultaneously.

If a user sends two messages within the same millisecond, IDs collide, causing React key collisions and undefined rendering. Use a counter or `crypto.randomUUID()`.

## Positioning conflict between chat FAB and ad-block banner

`SoftAdBlockMessage` renders at `fixed bottom-4 right-4 z-50` with `max-w-sm`. The chat floating button renders at `fixed bottom-6 right-6 z-40` (a 14x14 circle). Because the banner has higher z-index (50 vs 40), it obscures the chat button when both are visible. The banner only appears once per session and only when an ad blocker is detected, so the collision is intermittent, but when it happens the user cannot reach the chat toggle.

The open chat panel (`bottom-20 right-6 z-40`) would also sit beneath the banner's z-50 if both were visible simultaneously.

Fix options: raise the chat widget to z-50 and offset the banner upward, or dismiss the banner when the chat opens.

## Botpress loader left behind

The layout still renders `<BotpressLoader botConfigUrl={process.env.NEXT_PUBLIC_BOTPRESS_BOT_URL} />` outside `<Providers>`. If someone ever sets that env var in production, two chat widgets would appear simultaneously ‚Äî the Botpress webchat and the new floating widget. The import and render should be removed.

## Message ID generation

```tsx
const userMessage: Message = {
  id: Date.now().toString(),
  ...
};
// later
const assistantMessage: Message = {
  id: (Date.now() + 1).toString(),
  ...
};
```

The `+1` offset avoids collision between a user message and its immediate response, but doesn't protect against rapid consecutive sends. Two user messages in the same millisecond share the same ID, causing React key collisions and undefined rendering behavior.
