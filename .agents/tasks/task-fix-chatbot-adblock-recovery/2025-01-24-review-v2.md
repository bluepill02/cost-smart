# Ad-block recovery and floating chat widget (v2)

The change fixes `SoftAdBlockMessage` (imported but never rendered) by placing it inside the `<Providers>` wrapper where it has access to `useProStatus`, and adds a self-contained floating chat widget that replaces the non-functional Botpress integration when no bot URL is configured. The v2 revision addresses three of four v1 concerns: the chat button is repositioned to `bottom-24` to avoid obscuring the ad-block toast, message IDs now use a stable incrementing counter via `useRef`, and `FloatingChatWidget` is conditionally rendered only when `NEXT_PUBLIC_BOTPRESS_BOT_URL` is falsy.

**Watch for:** BotpressLoader still renders alongside the new widget — dead code that a future env var change could accidentally activate (confirmed); no test coverage for either component (confirmed); chat panel on mobile uses `max-sm:inset-4` which overrides `bottom-[7.5rem]`, covering the ad-block toast on small screens (likely).

## High-level view

The floating widget conditionally renders based on the absence of `NEXT_PUBLIC_BOTPRESS_BOT_URL`. This is a server-side check in a server component, so the widget's JS bundle is still shipped to the client regardless — it just won't mount. The BotpressLoader remains in the tree and is inert when the env var is empty. The two chat systems can never coexist at runtime, which resolves the v1 dual-widget concern functionally, though the dead code is still present.

The v1 ID collision and positioning overlap issues are resolved. The positioning fix moves the FAB to `bottom-24` and the panel to `bottom-[7.5rem]`, both at z-50. On desktop, the vertical separation from the toast at `bottom-4` is sufficient. On mobile, the panel's `max-sm:inset-4` overrides the desktop bottom offset and lands at the same position as the ad-block toast.

## BotpressLoader retention

The layout still contains:

```tsx
<BotpressLoader botConfigUrl={process.env.NEXT_PUBLIC_BOTPRESS_BOT_URL} />
```

The conditional render of FloatingChatWidget (`!process.env.NEXT_PUBLIC_BOTPRESS_BOT_URL && ...`) means the two never render simultaneously. But BotpressLoader's early-return on `!botConfigUrl` is the only guard — if the env var is set to any truthy string (even an invalid URL), Botpress scripts get injected while FloatingChatWidget disappears. There's no validation that the URL is actually reachable. Removing BotpressLoader entirely would make the architecture clearer now that a replacement exists.

## Mobile panel positioning overlap

The chat panel uses responsive overrides:

```tsx
className="fixed bottom-[7.5rem] right-6 z-50 w-[380px] ... max-sm:inset-4 max-sm:bottom-4 max-sm:right-4 max-sm:w-auto max-sm:max-h-none max-sm:fixed"
```

On viewports below the `sm` breakpoint, `inset-4` sets all edges to 1rem, then `bottom-4` overrides bottom to 1rem. SoftAdBlockMessage also renders at `fixed bottom-4 right-4 z-50`. When both are visible on mobile (ad blocker detected + chat open), they occupy the same z-layer and corner. The chat panel is larger and will visually cover the toast, but stacking between same-z siblings depends on DOM order — the toast renders before the widget in the tree, so the widget wins. The toast becomes unreachable while the chat is open on mobile.

Fix: raise the mobile panel's bottom to `max-sm:bottom-16` or dismiss the toast when the chat opens.

## Test gap

The v1 review flagged the absence of tests. This remains unaddressed. `analyzeQuery` and `generateResponse` are pure functions at module scope — they could be extracted to a shared module and unit-tested with minimal setup.
