# Comprehensive SEO fix: domain centralization and schema expansion

This change replaces all hardcoded staging URLs (`cost-smart-five.vercel.app`) with environment-driven configuration via a single `CANONICAL_DOMAIN` constant, expands JSON-LD schema coverage from 3 calculator pages to 33+, adds OG tags to pages missing them, injects Article schema into editorial blog posts, and re-targets H1/title keywords on two calculators.

**Watch for:**
- 7 OG `url` values were changed to relative paths (`'/in/emi-calculator'`) instead of absolute URLs â€” OG protocol requires fully-qualified URLs (confirmed)
- Article schema's `datePublished` defaults to `new Date().toISOString()` at render time, meaning it changes on every build/request rather than reflecting actual publication date (confirmed)
- `scripts/update_sitemap.ts` hardcodes `'https://costsmart.co'` instead of importing `CANONICAL_DOMAIN`, diverging from the centralization pattern (confirmed)

## High-level view

The domain centralization reads from `NEXT_PUBLIC_SITE_URL` with a fallback to `costsmart.co`. Non-source artifacts (audit CSVs, `seo-fix.js`, `DEPLOYMENT_CHECKLIST.md`) still reference the old domain but these aren't production code.

The schema expansion introduces a `CalculatorSchemaInjector` component that bundles `SoftwareApplication` + `FAQPage` schema, keyed by calculator type. The trade-off is that all calculators of the same type share identical generic FAQ content â€” Google may de-prioritize FAQ rich results when it sees duplicate content across pages.

The canonical URL strategy is inconsistent: some pages use relative paths for `alternates.canonical` (resolved by Next.js against `metadataBase`), while others use `${CANONICAL_DOMAIN}/path`. OG URLs have a hard requirement for absolute URLs per the Open Graph protocol, and 7 pages violate this.

The Article schema on editorial blog posts uses runtime `new Date()` as the `datePublished` default, so Google sees every blog post as "published today" on each crawl.

## Relative OG URLs break social sharing previews

The Open Graph protocol specification requires `og:url` to be an absolute URL. Seven pages in this diff set the OG URL to a relative path:

```tsx
// app/in/fd-calculator/page.tsx (and 6 others)
openGraph: {
  url: '/in/fd-calculator',  // Should be `${CANONICAL_DOMAIN}/in/fd-calculator`
}
```

When crawlers (Facebook, Twitter, LinkedIn) encounter a relative OG URL, behavior is undefined â€” some will ignore the tag, others may construct a malformed URL. These same pages already `import { CANONICAL_DOMAIN }` but only use it for other purposes. The fix is mechanical: prefix the path with `CANONICAL_DOMAIN`.

The remaining pages in this PR correctly use `` `${CANONICAL_DOMAIN}/path` `` for OG URLs, so this is an inconsistency within the same changeset.

## Article schema datePublished emits unstable values

All 8 editorial blog posts call `getArticleSchema()` without passing `datePublished`:

```tsx
<JsonLd data={getArticleSchema({ headline: '...', description: '...', urlPath: '/blog/ctc-vs-in-hand' })} />
```

The fallback in `lib/seo-utils.ts` is:

```ts
datePublished: props.datePublished || new Date().toISOString(),
dateModified: props.dateModified || new Date().toISOString(),
```

On every SSR render or static build, this produces a different timestamp. Google Search Console may flag this as unstable structured data. Either pass real dates or omit the field entirely â€” a missing `datePublished` is better than a lying one.

## scripts/update_sitemap.ts diverges from centralized domain

The runtime `app/sitemap.ts` correctly uses `CANONICAL_DOMAIN`, but the code-generation script hardcodes the domain:

```ts
// scripts/update_sitemap.ts
const content = `...
  const baseUrl = 'https://costsmart.co';
...`;
```

If this script is re-run, it will overwrite `app/sitemap.ts` with a version that doesn't import `CANONICAL_DOMAIN`. The script should emit code that imports the constant.

## Redirect page metadata conflict

`app/in/home-loan-calculator/page.tsx` immediately `redirect()`s to `/home-loan-calculator`. Its metadata sets:
- `canonical: '/home-loan-calculator'` (the redirect target)
- `openGraph.url: '${CANONICAL_DOMAIN}/in/home-loan-calculator'` (the page that redirects)

Since the page redirects, search engines shouldn't index it, but if they do encounter the metadata, the OG URL contradicts the canonical signal. Either remove OG tags from redirect pages or point OG URL at the canonical target.

## Duplicate FAQ content risk across calculator types

Every "business" calculator (Break-Even, GST Input Credit, Invoice Generator, Shipping, Freelance Rate) shares the same 4 FAQ questions via `CalculatorSchemaInjector`. Google may de-prioritize FAQ rich results when it detects identical FAQ content across many pages on the same domain. No FAQ information was lost during migration from the previous per-page approach â€” the same `DEFAULT_FAQS` map was already in use â€” but the expansion from 3 pages to 33+ amplifies the duplication signal.
