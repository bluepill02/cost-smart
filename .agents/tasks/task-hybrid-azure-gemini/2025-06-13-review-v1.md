# Hybrid AI split: Azure AI Services for Language/Vision, Gemini for generative

Five API routes (translate, detect-language, document-scan, document-analyze, financial-health) are reverted from Gemini to Azure AI Services REST APIs. A new `lib/azure-ai.ts` module provides authenticated helpers for Translator, Computer Vision Read, and Text Analytics. Two generative routes (ai-insights, ai-advisor) remain on Gemini via Vertex AI. Graceful fallback logic is added to translate and detect-language so they return degraded but non-error responses when Azure is unreachable.

**Watch for:** Azure credentials return 401 during testing (confirmed) — all five Azure routes currently operate in fallback-only mode. The document-scan route has no fallback at all and returns 502 on Azure failure (confirmed). The `isRealAI` response field communicates whether the client received an AI-backed result, but only two of the five Azure routes expose it (likely gap).

## High-level view

The central abstraction is `lib/azure-ai.ts` — a thin REST wrapper over four Azure endpoints (translate, detect, read/OCR, sentiment, key phrases) using fetch and `Ocp-Apim-Subscription-Key` auth. Routes import individual functions instead of sharing a Gemini client, which cleanly separates the provider boundary.

Fallback strategy is inconsistent across the five routes. Translate and detect-language catch Azure errors and serve local heuristic results. Document-analyze and financial-health catch errors and fall back to local keyword extraction. Document-scan has no fallback and fails hard with 502. A user hitting any of these routes today will get fallback behavior on translate/detect/analyze/health and a hard failure on document-scan.

The OCR route (document-scan) uses Azure Computer Vision Read, an async polling API. The implementation polls once per second for up to 30 seconds, which means a single request can hold a serverless function alive for 30 seconds on slow or large documents. There is no abort signal or early timeout.

## Fallback asymmetry across routes

Three distinct fallback strategies exist:

1. **Translate / detect-language**: catch Azure errors, serve a degraded response (original text / regex heuristic), flag `isRealAI: false`.
2. **Document-analyze / financial-health**: catch Azure errors, fall back to local keyword-based classification. No `isRealAI` flag is returned.
3. **Document-scan**: no fallback. Azure failure returns 502 to the client.

The asymmetry means the user experience degrades differently per tool when the Azure key is invalid. Document-scan is the worst case: it's the only route that becomes completely unusable without a working Azure endpoint, since OCR has no heuristic approximation.

For document-analyze and financial-health, the catch block falls back to empty `keyPhrases` and local classification. The fallback in financial-health is slightly richer: it scans for positive/negative phrases and synthesizes a sentiment score. Document-analyze's fallback produces no key phrases at all and relies entirely on the document-type classifier.

## OCR polling lifecycle

`azureReadOCR` submits a binary image, reads the `Operation-Location` header, then enters a polling loop:

```
for (let attempt = 0; attempt < 30; attempt++) {
    await new Promise(resolve => setTimeout(resolve, 1000));
    // poll GET on operationLocation
}
```

This is 30 seconds worst-case wall-clock time within a single request handler. On Vercel Hobby (10s timeout) this will always fail for slow documents. On Pro (60s) it fits, but the function is billable for the entire poll duration. There's no `AbortSignal` integration, so if the platform kills the function mid-poll, the fetch may leave dangling connections.

An exponential backoff starting at 500ms (doubling to 2s, 4s) would reduce unnecessary polls for fast responses while still reaching the same total timeout.

## Translate route: financial glossary becomes post-processing only

The previous Gemini implementation could include glossary terms in the prompt to guide domain-specific translation. The Azure Translator integration no longer has a mechanism to inject glossary hints into the translation itself — the `FINANCIAL_GLOSSARY` dictionary is returned alongside the translation for client-side reference, but doesn't influence translation quality. This is a functional regression in domain-specific accuracy for financial terms: Azure Translator will translate "EMI" literally rather than preserving it as an acronym with a parenthetical explanation.

Azure Translator does support custom dictionaries via the Custom Translator portal or dynamic dictionary markup in the request body (`<mstrans:dictionary>` tags). Neither is used here.

## Credential failure mode

Testing confirmed all five Azure endpoints return 401 with the provided key. Possible causes: the multi-service resource may not have Translator enabled (requires explicit opt-in on some SKUs), the key may have been regenerated, or the `uaenorth` region may not support all services under this resource. Until resolved, the application runs entirely on fallback logic for Azure-backed features.
