# Neighborhood Explorer with Google Places integration

A new freemium feature that lets users explore nearby amenities (schools, hospitals, parks, transit, shopping) for any location. Two API routes handle place search and geocoding with rate limiting and server-side Pro verification. A client component provides address/geolocation input, displays categorized results with a composite Neighborhood Score, and gates detailed data behind a Pro paywall. CTA widgets drive traffic from existing calculator pages.

**Watch for:** Hardcoded API key committed to `.env.local` and visible in FEAT-001.json task description (confirmed). The `NEXT_PUBLIC_GOOGLE_MAPS_API_KEY` is embedded in client-rendered iframes on both the Neighborhood Explorer and all solar-roi city pages, meaning anyone can extract it from page source (confirmed). The explore route makes 5 sequential Google Places API calls per user request with no parallelization (confirmed) — this creates 2-5 second response times and amplifies cost per free-tier request.

## High-level view

The API layer follows the established solar-insights pattern: in-memory rate limiting (acknowledged as best-effort on serverless), server-side Pro verification via KV store, and tiered response shaping. The explore route fires 5 sequential Google Places Text Search requests per user lookup, each costing against the project quota, with only 2 free requests/day/IP as the guardrail.

The client component mirrors SolarAnalyzer's UX: Nominatim geocoding first, fallback to the paid geocode route, browser geolocation, manual coords. The Pro paywall gates extended place details behind the familiar blur overlay, enforced server-side (free responses strip ratings/addresses).

The `NEXT_PUBLIC_` key is an architectural constraint of Google Maps Embed — the key must be in the iframe URL. But the same key value is shared across server-side and client-side env vars, meaning if the client-visible key isn't referrer-restricted in Google Cloud Console, it grants access to the Places and Geocoding APIs directly.

## Sequential upstream calls and cost amplification

The explore route iterates over 5 categories in a `for` loop, awaiting each `fetch` sequentially:

```typescript
for (let i = 0; i < CATEGORIES.length; i++) {
  const placesResponse = await fetch('https://places.googleapis.com/v1/places:searchText', ...);
  // ...
}
```

Each request takes 200-500ms typically, so a single explore call takes 1-2.5 seconds in the happy path. The requests are independent — replacing this with `Promise.allSettled` would preserve graceful degradation (empty results for failed categories) while cutting wall-clock time by ~4x.

From a cost perspective, each free-tier explore request generates 5 Places API calls. At 2 free requests/day/IP, a single IP costs 10 Places calls/day. If the in-memory rate limiter resets across serverless cold starts (which it will), effective limits could be higher in practice — the Google Cloud per-key quota is the real backstop.

## Neighborhood Score edge case with locationBias

The scoring algorithm grants up to 20 points per category (10 for count of results, 10 for proximity). The `locationBias` in the Places API request is a bias, not a hard filter — Google can return places outside the specified radius. When this happens, `(1 - avgDistance/maxRadius)` goes negative. The `Math.max(0, ...)` guard prevents negative scores, but categories with far-flung results will silently contribute 0 proximity points, making the score harder to interpret. Worth documenting or clamping results to within-radius places.

## Wasted Maps Embed API call for free users

The Google Maps Embed iframe in NeighborhoodExplorer is inside the blur overlay div but still rendered in the DOM for free users:

```tsx
<div className={!isPro ? 'blur-sm pointer-events-none select-none' : ''}>
  <iframe src={`...key=${process.env.NEXT_PUBLIC_GOOGLE_MAPS_API_KEY}...`} />
</div>
```

The iframe loads and makes an API call even when visually hidden behind the paywall overlay. Conditionally rendering the iframe only when `isPro` is true would avoid an unnecessary Maps Embed API call per free-user lookup.
