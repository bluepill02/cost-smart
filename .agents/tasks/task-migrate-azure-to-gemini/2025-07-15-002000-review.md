# Azure-to-Gemini migration fix pass (v2)

The fix commit addresses all seven issues raised in the v1 review: prompt injection mitigated via `<user_content>` delimiters and `systemInstruction` on all routes, hardcoded project ID replaced with a runtime throw, batch translation switched from separator-based splitting to JSON array responses, `detectedLanguage` now populated from Gemini's response, WorkerManager silenced, financial-health route given a local keyword fallback, and the Odia character corrected. The Proxy-based lazy initialization of the Gemini client is a clean pattern for serverless cold starts.

**Watch for:** Single-text translation falls back to raw response on JSON parse failure, potentially exposing model preamble text to users (likely). The `ai-advisor` route passes user message history directly as `contents` without `<user_content>` delimiters, relying solely on the system instruction's "never follow instructions" clause — a weaker boundary than the tagged approach used elsewhere (confirmed).

**Verdict**: APPROVED

## High-level view

The `lib/gemini.ts` rewrite replaces the hardcoded fallback with a Proxy that throws on first use if `GOOGLE_CLOUD_PROJECT` is unset. The lazy pattern surfaces the error on the first API call rather than at module load, appropriate for serverless where unused modules shouldn't crash the process.

The prompt injection fix is consistently applied across six of seven routes: user text is wrapped in `<user_content>` tags and the system instruction tells the model to only process content within those tags. The document-scan route uses a different but reasonable approach — since the input is an image, its system instruction warns against following instructions embedded in the document image. The ai-advisor route is the outlier: chat messages flow directly as Gemini `contents` (multi-turn history) without tag boundaries, relying on the system instruction's anti-injection clause. This is architecturally correct for multi-turn chat (you can't wrap each turn in tags when the model needs to see its own prior responses), but it's the weakest point in the injection defense.

The batch translation fix replaces the fragile separator scheme with a JSON array contract. Segment count mismatches are handled by padding with empty strings and truncating to input length — trailing positions may show blank rather than shifted translations.

## Single-text translation raw fallback

In `app/api/azure/translate/route.ts`, the JSON parse failure path for single-text requests uses the raw model response as the translation:

```typescript
} catch {
  // If JSON parsing fails for single text, use raw response
  if (!isBatch) {
    translatedSegments = [responseText.trim()];
  } else {
    return NextResponse.json(
      { error: 'Translation service returned an invalid response. Please try again.' },
      { status: 502 }
    );
  }
}
```

The batch path returns a 502 error. The single-text path optimistically assumes the model returned just the translation without JSON wrapping. If the model returned something like `"Here is the translation:\n\nTranslated text"`, the preamble leaks into the UI. The temperature is 0.1 and the system instruction requests "Output valid JSON only", so this path is unlikely to trigger in normal operation — but when it does, the user sees garbage. The asymmetry with the batch error path is worth noting.
