# Azure AI multi-service integration (v2)

This change adds four Azure-powered features to CostSmart: receipt OCR scanning, multilingual financial translation, a financial health score analyzer, and pricing/premium page updates. The v1 review's seven issues have been largely addressed — credentials moved to env vars, rate limiting added to detect-language, privacy copy updated, polling reduced to 8 seconds, image size validated server-side, FinancialHealthAnalyzer reads subscription from localStorage, and all routes use `process.env`. One component was missed in the subscription fix.

**Watch for:** The `FinancialTranslator` component still hardcodes `isPro = false` and passes `subscriptionId: undefined`, meaning Pro users can never access premium languages or batch translation from this component (confirmed). The `TranslateButton` component similarly never passes subscription credentials, though its design as a free-tier quick-translate widget may be intentional (likely).

## High-level view

Six of seven v1 issues are fully resolved. All API routes now read credentials from `process.env.AZURE_AI_KEY` and `process.env.AZURE_AI_ENDPOINT`. The detect-language endpoint has per-IP rate limiting (20 req/min) with a 1,000-character input cap. The tools page privacy copy now accurately distinguishes between local calculators and cloud-powered AI tools. The OCR polling loop is capped at 8 attempts (8 seconds total), fitting within Vercel function timeouts. Server-side image size validation rejects payloads over 10MB before buffering.

The one remaining subscription gap is in `FinancialTranslator.tsx`, which hardcodes `isPro = false` and never reads localStorage. This is the same pattern that was fixed in `FinancialHealthAnalyzer.tsx` and `DocumentScanner.tsx` but was not applied to the translator component. Pro subscribers attempting to translate to Telugu, Bengali, or other premium languages will be blocked by the server's free-tier enforcement.

The `TranslateButton` component (a floating widget designed for embedding on calculator pages) also sends no subscription info, but it offers all 10 languages in its UI with no free/Pro distinction. When a Pro user selects a premium language through this button, the server will reject the request with a 403.

## FinancialTranslator subscription gap

The `FinancialTranslator.tsx` component declares:

```typescript
const [isPro] = useState(false);
```

And later:

```typescript
body: JSON.stringify({
  text: sourceText,
  targetLanguage: targetLang,
  isPro,
  subscriptionId: undefined,
}),
```

Compare with the fixed `FinancialHealthAnalyzer.tsx` which now reads:

```typescript
const subscriptionId = typeof window !== 'undefined' ? localStorage.getItem('subscriptionId') : null;
const userIsPro = !!subscriptionId;
```

The translator component additionally performs a client-side gate check (`if (!lang?.free && !isPro)`) that blocks the request before it even reaches the server, so a Pro subscriber will see "This language requires a Pro subscription" despite having paid. The server-side verification is correctly implemented in the translate route — only the client component fails to supply credentials. **Confirmed** — lines 38 and 67-68 of FinancialTranslator.tsx.

## TranslateButton language/credential mismatch

The `TranslateButton` component renders all 10 Indian languages (no free/Pro distinction in its LANGUAGES array) and sends a minimal request body:

```typescript
body: JSON.stringify({
  text,
  targetLanguage: targetLang,
}),
```

No `isPro` or `subscriptionId` fields are included. The translate API route treats missing `isPro`/`subscriptionId` as a free-tier request, which means only Hindi and Tamil will work. Users selecting any other language from the button's dropdown will receive a "Free tier only supports Hindi and Tamil" error. Since this button is designed to be embedded across calculator pages, it may be acceptable for it to be free-tier-only — but then the UI should only show Hindi and Tamil, not all 10 languages. **Confirmed** — TranslateButton.tsx line 43-47.
