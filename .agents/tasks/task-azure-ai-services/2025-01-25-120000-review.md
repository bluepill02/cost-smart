# Azure AI multi-service integration

This change adds four Azure-powered features to CostSmart: receipt OCR scanning, multilingual financial translation, a financial health score analyzer, and pricing/premium updates to showcase them. Each feature gets an API route, a client component, and a tool landing page with SEO metadata. The architecture follows the existing pattern of server-side Pro verification via PayPal subscription checks, with free tiers offering degraded results.

**Watch for:** Hardcoded Azure API key in every route file (confirmed) — this is a live credential committed to the repository. The tools page claims "no data is stored or sent to servers" while every new tool sends user data to Azure (confirmed). The `detect-language` route has no Pro gating or rate limiting at all (confirmed).

## High-level view

The Azure API key was initially hardcoded as a string literal in five route files but has been fixed to use `process.env.AZURE_AI_KEY`. The `.env.example` defines `AZURE_AI_ENDPOINT` and `AZURE_AI_KEY` variables which are now referenced by all routes.

Pro gating is enforced server-side for the main features (scan, translate, health) but the `detect-language` endpoint is completely ungated — no subscription check, no character limit beyond 10,000, no rate limiting. This endpoint could be called in a loop to run up Azure costs at will.

The trust badge on the tools listing page (`app/tools/page.tsx` line 157) says "All tools run in your browser — no data is stored or sent to servers." Every new tool sends images, financial text, or translation content to Azure via the app's own API routes. This is misleading to users.

The document scanner's polling loop (`pollForResults`) awaits up to 30 seconds with 1-second intervals inside the request handler. Next.js serverless functions on Vercel have a 10-second timeout on the Hobby plan, making this likely to fail in production. The polling has no abort mechanism if the client disconnects.

## Credential exposure across all Azure routes

Every new route file reads the API key from environment variables:

```typescript
const AZURE_API_KEY = process.env.AZURE_AI_KEY || '';
```

This appears in `detect-language/route.ts`, `document-analyze/route.ts`, `document-scan/route.ts`, `financial-health/route.ts`, and `translate/route.ts`. The project's other routes (Google Maps, Solar, OpenAI, Resend) all use `process.env.*` and the `.env.example` even defines the correct variable names for Azure. The key needs to be rotated — removing it from code doesn't remove it from git history. **Confirmed** by reading all five route files.

## Unprotected language detection endpoint

`app/api/azure/detect-language/route.ts` validates that `text` is present and under 10,000 characters, then directly calls Azure Translator's detect endpoint. Unlike every other Azure route, it has no subscription verification, no `isPro` check, and no free-tier restriction. Any client can call it in an unbounded loop. The translate route enforces a 500-character free limit and language restrictions; the detect endpoint should at minimum share similar constraints. **Confirmed** — the route has zero gating code.

## OCR polling in a serverless function

The document scan route calls Azure's async Read API, then enters `pollForResults`:

```typescript
async function pollForResults(operationUrl: string): Promise<string> {
  const maxAttempts = 30;
  const delay = 1000;
  for (let i = 0; i < maxAttempts; i++) {
    await new Promise(resolve => setTimeout(resolve, delay));
    // ...
  }
}
```

This blocks the request handler for up to 30 seconds. Vercel's default function timeout is 10 seconds (Hobby) or 60 seconds (Pro). On Hobby, the OCR feature will consistently time out. There's also no mechanism to detect client disconnect — if the user navigates away, the function continues polling and consuming resources until it completes or times out. **Confirmed** — the polling is synchronous within the POST handler with no streaming or early-exit mechanism.

## FinancialHealthAnalyzer Pro subscription never connected

The client component `FinancialHealthAnalyzer.tsx` declares:

```typescript
const [isPro] = useState(false);
```

This is never updated from localStorage or any subscription context. Compare with `DocumentScanner.tsx` which reads `localStorage.getItem('subscriptionId')` before making the API call. The health analyzer always sends `isPro: false` and `subscriptionId: undefined`, so even paying Pro users will see the gated free-tier response (score only, no recommendations). **Confirmed** — searching the component reveals no subscription read logic.

## Server-side input validation gap for document-scan

The client validates image type and 10MB limit before sending. The server only checks `if (!image)` — it doesn't validate that the base64 payload decodes to a valid image or that its size is within bounds. A direct API call could send a 100MB base64 string that would be buffered entirely into memory as a Node.js Buffer before being rejected by Azure. **Confirmed** — the route contains only a truthy check on the `image` field.

## Misleading trust copy contradicts new features

`app/tools/page.tsx` line 157:

```
All tools run in your browser — no data is stored or sent to servers.
```

The new document scanner sends base64 images to the backend which forwards them to Azure. The translator sends text to Azure Translator. The health analyzer sends financial descriptions to Azure Text Analytics. This statement was perhaps accurate when the tools were client-only calculators but is now false. **Confirmed** — three of the five listed tools send data to external cloud services.
