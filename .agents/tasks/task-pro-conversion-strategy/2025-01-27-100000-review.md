# Pro conversion funnel: gate modal, analytics preview, comparison pages, social proof

This change builds the conversion infrastructure for Cost-Smart's Pro tier. It introduces a feature-gating system (UpgradeGateModal, ProBadge, ProFeatureWrapper) backed by a centralized feature registry, adds blurred analytics previews and insight teasers to the dashboard, creates SEO-oriented comparison pages (vs Excel, vs YNAB) and persona landing pages (freelancers, budget-savvy), and adds a testimonials section and email-gated budget template resource page.

The approach is purely frontend: gating relies on `useProStatus()` which already exists in the codebase, and the new components render blurred/locked states when the user isn't Pro. No server-side enforcement is added. The comparison and persona pages are static marketing content with structured data and ad placements.

Watch for: The feature gate is cosmetic-only — blurred CSS with `pointer-events: none` offers no actual access control (confirmed). The budget template email gate captures an address then immediately hands the user the download link client-side, so the PDF is publicly accessible at `/budget-template.pdf` regardless (confirmed). The package-lock.json changes are unrelated Azure SDK additions from a different task (confirmed).

## High-level view

The gating system (`ProFeatureWrapper` → `UpgradeGateModal`) uses CSS blur and a client-side `isPro` check to hide premium content. Since all chart data in `ProAnalyticsPreview` is hardcoded mock data, there's no real content to protect, but the pattern establishes a convention for future gating. Any feature that delivers actual value (PDF export, AI advice) still needs server-side enforcement in its respective API route — the gate is a conversion nudge, not access control.

The comparison pages (vs Excel, vs YNAB) and persona landing pages hardcode the Pro price as "$4.99/month" in multiple places rather than referencing `PREMIUM_CONFIG`. If pricing changes, these pages will show stale numbers until manually updated across 10+ occurrences.

The budget template resource page uses an email capture form that posts to the existing `/api/lead-capture` endpoint, then reveals a download link to `/budget-template.pdf`. The PDF itself is a two-line placeholder text file committed to `public/`, accessible to anyone who navigates directly to the URL.

The testimonials component renders hardcoded reviews on both the homepage and pricing page. All ratings are 5 stars with no variation, and both pages show the identical set — a user navigating from home to pricing sees the same social proof twice.

## Client-only gating via ProFeatureWrapper

`ProFeatureWrapper` checks `useProStatus()` and either renders children directly (Pro users), shows a loading state (opacity 50%), or wraps children in a `blur-[2px] opacity-60 grayscale` container with `pointer-events-none`. The children are still rendered in the DOM — they're just visually obscured. A user can remove the CSS classes in devtools to reveal the content underneath. For `ProAnalyticsPreview` this is harmless because the data is static mock, but any new developer might assume wrapping a feature in `ProFeatureWrapper` is sufficient protection when it is not.

Both `ProBadge` and `ProFeatureWrapper` independently instantiate their own `UpgradeGateModal`. Clicking the badge inside the wrapper's overlay could trigger two modals if event propagation isn't stopped. The badge does call `e.stopPropagation()` which handles this, but the relationship is fragile — both components manage their own `isOpen` state independently, and a future refactor that removes the `stopPropagation` call would produce a double-modal bug. (likely)

## Budget template email gate bypass

`BudgetTemplateForm` submits to `submitLeadCapture()` (fire-and-forget POST to `/api/lead-capture`), then flips `submitted` state to true, revealing a download link to `/budget-template.pdf`. The PDF lives in `public/` and is served statically by Next.js with no auth middleware. Bots and anyone who views page source will find the URL immediately. If meaningful gating is desired, serve the PDF through an API route that checks a one-time token generated on successful form submission.

The committed file at `public/budget-template.pdf` contains:
```
CostSmart Monthly Budget Template - Placeholder
This is a placeholder file for the budget template PDF download.
```

This is plain text, not a valid PDF. Any user who completes the form and clicks download will receive a broken file.

## Comparison pages hardcode pricing in 10+ locations

Both comparison pages (`cost-smart-vs-excel/page.tsx`, `cost-smart-vs-ynab/page.tsx`) embed "$4.99/month" in quick-summary cards, comparison tables, verdict sections, and CTAs. The YNAB page additionally states "Save $120/year" which is arithmetically correct only at these exact price points. The codebase already has `PREMIUM_CONFIG` as a pricing source of truth, but these pages don't import from it. A pricing change requires a find-and-replace across both 400-line files.

## Testimonials rendered identically on two pages

`TestimonialsSection` is placed on both the homepage (between the calculator grid and Pro value prop) and the pricing page (between plan cards and FAQ). The same 4 testimonials with the same 5-star ratings appear in both locations. A user following the conversion funnel (home → pricing) sees identical content, which diminishes its persuasive effect. Consider different testimonial subsets per page, or at minimum introduce rating variation to appear more authentic.
